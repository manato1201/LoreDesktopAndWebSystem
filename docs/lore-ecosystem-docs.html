01 / Overview

架空の巨大バイナリVCS「Lore」を中心としたエコシステム

GitHubに相当するLoreHub(Web)、Forkに相当するLoreForge Client(デスクトップ)、Docker Desktopに相当するLoreForge Server Admin(デスクトップ)の3つを、単一のデザイントークンと単一のRustバックエンドで束ねている。

ローカルでのリポジトリ操作・ファイルロック・プレビュー
Desktop

LoreForge Server Admin

Loreサーバー環境をGUIから1クリック構築・権限設計

04 / Data Flow

認証シーケンス

Server Componentはブラウザとは別プロセスのため、受信したCookieを手動でAPIリクエストへ転送する必要がある — この非対称性がこのアーキテクチャの要点。

Browser → Web
/login で email / password を送信(Client Component)
Web → API
POST /api/auth/login(credentials: include)
API
argon2でパスワード検証 → セッション発行
Set-Cookie: lorehub_token(HttpOnly, SameSite=Lax, 7日)
Browser → Web(SSR)
以降のページ遷移: GET /repositories/xxx
Web(Server Component)
cookies() からセッションCookieを取得
ブラウザのCookieジャーとは無関係な別プロセス
Web → API
Cookieヘッダを明示的に転送して再フェッチ
これを忘れると SSR は常に401になる

10 / Workflow

マルチエージェント並行開発

ディレクトリが重ならない3つの作業を分割し、バックグラウンドサブエージェントとして並行実行した。

Main session
lorehub-web/ — リポジトリ設定機能(直接実装)
Main session
loreforge-client/ — コミット履歴ビュー(sub-agent)
Main session
loreforge-server-admin/ — 権限設定永続化(sub-agent)

鍵となったのは「スコープの分離」(ディレクトリが衝突しない)と「知識の伝播」(1体目が発見した罠 — QML singleton、SendInputの不通、QStringLiteralの制約など — を後続のプロンプトに引き継いだこと)。

11 / Trust

独立再検証パターン

Phase 7ではサブエージェントの完了報告を鵜呑みにせず、オーケストレーターが毎回ゼロから独立して再現するルールを徹底した。

Step 1
build/ を完全削除してクリーンビルド
Step 2
報告内容を見ずに、報告された動作を自分の手で再現
Step 3
一次証拠を取得
実プロセスのPID・tasklist出力・HTTPレスポンスの実バイト数・Windowsレジストリの値そのもの
Step 4
TEMP-VERIFY タグの検証用コードが完全除去されているか grep で確認

この過程で、報告だけでは分からない環境ドリフト(Visual Studioのインストールパスがセッション中に 2022 フォルダから 18 フォルダへ自動更新されていた)や、検証手法自体の欠陥(QMLの console.log がリダイレクトログへ確実にフラッシュされない、複雑な画面遷移よりC++側の単機能を直接検証する専用QMLの方が確実)を発見した。
